# 关于 v2 修订

> 这是一份给读者的说明：本书为什么有 v2、v2 比 v1 多了什么、之前看过 v1 的读者要不要重看。

## 一句话版本

**源码没变，模型变了。** v2 用更强的新一代模型重新通读了同一份 Claude Code 源码，发现了 v1 漏讲的一整片模块，纠正了一些数字错误，并把全书的组织方式从"模块树"换成了"一次会话从启动到产出的完整链路"。

## 为什么会有 v2

v1 是 2026 年 4 月用 Claude Opus 4.6 解析、ChatGPT 5.4 评审完成的，共 25 篇。

写完 v1 之后，模型升级到了 Claude Opus 4.7 和 ChatGPT 5.5。同样的一份 Claude Code 源码，新模型读出了 v1 没讲清楚甚至完全没注意到的内容：

- **远程会话**：本地 CLI 怎么被手机/浏览器接管（v1 完全空白）
- **企业代理**：CLI 在公司内网里怎么连上游 API（v1 完全空白）
- **Coordinator 与 Cron**：没人按回车时会话怎么继续转（v1 完全空白）
- **Buddy 宠物 / Vim 模式 / Voice 输入 / Doctor 屏 / Output Style**：这些"非主干"但实打实存在的子系统（v1 完全空白）
- **配置迁移**：产品在用户身后偷偷换零件的那 11 个迁移函数（v1 完全空白）

再加上 v1 里一些"约 1900 行"、"5+1 层配置"这类含糊或失真的数字，都需要按当前源码 commit 重新核对一遍。

所以 v2 不是把 v1 重写得更花哨，而是**用更强的模型把同一份源码重新扫一遍，把漏的补上、把错的改对、把组织方式调到读者更舒服的样子**。

## v2 相比 v1 的三个变化

### 1. 章数：25 篇 → 34 章

新增 8 章全新内容，全部基于 v1 当时未深扫的源码模块：

| 新章 | 讲什么 |
|---|---|
| 第 4 章 · 配置迁移即代码 | 11 个迁移文件如何让 settings 静默升级 |
| 第 13 章 · 通信、调度、问询与合成工具 | WebFetch、ScheduleCron、SendMessage、AskUserQuestion 等 10 个"对话外"的工具 |
| 第 17 章 · Coordinator、Cron 与定时调度 | 多 Agent 编排层 + 定时触发 |
| 第 24 章 · Bridge IPC 与远程会话 | 手机/Web/Desktop 控制本地 CLI 的全链路 |
| 第 25 章 · DirectConnect 与上游代理 | 企业内网代理拓扑 |
| 第 28 章 · Keybindings、Vim 模式与 Voice 输入 | 终端输入层的三种解释 |
| 第 29 章 · Buddy 宠物 | PromptInput 边上那只会眨眼的小动物 |
| 第 30 章 · Doctor 屏与 Output Style 体验 | 自检仪表 + 换装系统 |

剩下 26 章里：
- 11 章是「勘误保留」：v1 内容大体保留，只修正数字、行号、过期 API；
- 9 章是「迭代重写」：基于 v1 骨架，新增段落覆盖新发现的子模块；
- 3 章是「保留」：v1 写得没问题，照搬；
- 3 章是「拆分合并」：v1 一章里塞了两件事，拆开讲更清楚。

### 2. 组织方式：模块树 → 运行期生命周期

v1 是按"模块"平铺的——工具一篇、命令一篇、Agent 一篇……读者要自己拼凑"一次对话从开始到结束都经过了哪些模块"。

v2 改成 **8 篇**，按 Claude Code 实际运行时的链路展开：

```
第一篇 · 进入软件之前         (启动、配置、迁移)
第二篇 · 一次回合的内核        (对话循环、prompt、压缩、cache、thinking)
第三篇 · 工具家族              (协议、Bash、文件/LSP、通信调度)
第四篇 · Agent、任务与协调     (Agent、TaskType、Coordinator)
第五篇 · 协议、安全与扩展接口  (MCP、权限、Hooks、扩展点、feature flag)
第六篇 · 网络层与远程协作      (传输层、Bridge IPC、DirectConnect)
第七篇 · 终端 UI 与多模态输入   (Ink、设计系统、Vim/Voice、Buddy、Doctor)
第八篇 · 记忆、扩展与总结       (Memory、命令系统、状态管理、架构模式)
```

读者按顺序看下来，就是从你敲下 `claude` 开始，一直到 AI 给出答复并保存上下文的完整链路。

### 3. 数字与引用全部对齐到同一个源码 commit

v2 的所有"X 行 / X 个 / X 种"都重新对齐到了同一个冻结的源码 commit（`290fdc9`），并且关键的工具、命令、Hook 事件、内置 Agent、TaskType 都额外整理成了 6 份**附录速查表**，每张表都对应一份脚本生成的 manifest，可复核：

- 附录 A：工具速查表
- 附录 B：Commands 速查表
- 附录 C：Hooks 事件表
- 附录 D：内置 Agent 速查表
- 附录 E：TaskType 谱系
- 附录 F：模块 × 章节双向矩阵

## 之前看过 v1，需要重看吗？

不需要全部重看。三种情况：

**情况 1：你只想补 v1 漏掉的部分。** 直接读 v2 新增的 8 章——这些是 v1 完全没讲的内容。如果你写过 Claude Code 的插件、用过它的远程会话、好奇 Buddy 那只小动物是怎么实现的，这 8 章会让你看到完全没料到的设计。

**情况 2：你想把当年看 v1 的笔记对齐到最新源码。** 关注「迭代重写」与「勘误保留」两档章节里那些被修订的小节，重点是数字、行号、API 名称的变化。

**情况 3：你之前没读过 v1。** 直接读 v2，按 [00-目录与阅读指引](./00-目录与阅读指引.md) 推荐的路线进入即可，不需要先看 v1。v1 与 v2 是同一份源码的两次解读，v2 是更准确、更完整的那一版。

## 一点声明

v2 不是 v1 的"重写本"，而是 v1 的"修订本"。

对于 v1 已经写好的章节，v2 在改动时遵循**最小修改原则**：错的改对，缺的补上，文风、节奏、叙事方式尽量保留 v1 的样子。这是因为 v1 的写法本身是好的——"用第二人称带读者读源码"这种叙事博客的口吻就是这本书最初想做的事，v2 要做的是让事实更准，而不是让风格更"专业"。

所以你在 v2 里看到的大多数章节，文气还是 v1 的文气。改动主要发生在：被新模型识别出来的事实错位、被新源码倒推出来的概念变化、以及那 8 篇 v1 完全空白的新章节。
